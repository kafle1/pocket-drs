\begin{center}\textbf{\fontsize{16}{19}\selectfont ABSTRACT}\end{center}
\addcontentsline{toc}{chapter}{Abstract}
\vspace{0.5cm}

PocketDRS is a mobile-to-server Decision Review System that provides low-cost Leg Before Wicket (LBW) assistance for cricket using a single smartphone camera. Professional systems such as Hawk-Eye rely on multiple synchronized high-speed cameras and specialized hardware, making them unaffordable outside elite venues. PocketDRS demonstrates that a constrained single-view workflow can deliver practical decision support for schools, clubs, and training grounds. A Flutter application records a delivery and lets the user mark the four pitch corners; a FastAPI server detects the ball from motion and colour, associates the detections into one trajectory, recovers the camera pose from the marked corners, and reconstructs the ball path in real-world 3D using projectile-motion physics. A rule-based engine then applies the three LBW checks of ICC Rule 36. On 66 simulated deliveries the system recovered the camera position correctly every time and returned the correct verdict in 53 cases (80.3\%), including all 16 OUT cases. The system supports, but does not replace, on-field umpiring.

\chapter{Introduction}

\section{Introduction}

Cricket officiating increasingly relies on technology for reviewing close decisions, particularly Leg Before Wicket (LBW) cases. In elite matches, systems such as Hawk-Eye reconstruct the ball trajectory using multiple synchronized high-speed cameras and specialized processing infrastructure. Although highly effective, these systems are costly, complex to operate, and impractical for schools, clubs, local leagues, and training environments.

PocketDRS is proposed as a low-cost decision-support system that uses a single smartphone camera and a lightweight backend service to analyze a delivery. The user records or selects a short video clip, calibrates the visible pitch by marking known reference points, submits the delivery for processing, and receives a visual explanation together with an LBW assessment. The project emphasizes practicality over broadcast-grade precision, targeting a use case where approximate decision support is valuable even without ICC certification.

The primary innovation is demonstrating that a constrained single-view pipeline, leveraging known pitch geometry and projectile motion physics, can produce consistent and interpretable LBW review support. The current prototype demonstrates this through a working mobile-to-cloud system combining computer vision, geometrical reconstruction, and domain-specific decision logic.

\section{Problem Statement}

Professional DRS systems depend on multiple synchronized cameras to recover depth and reconstruct the ball trajectory in three dimensions. This produces strong results, but the required equipment, installation effort, and operating cost make such systems inaccessible outside major venues.

When the same decision-review problem is shifted to a single consumer camera, several technical and practical difficulties arise:

\begin{enumerate}
    \item A monocular view does not provide direct depth information, requiring alternative cues such as ball size or strong motion constraints.
    \item A cricket ball is small, fast, motion-blurred, and can easily be confused with background objects, players, shadows, and spectators.
    \item Small calibration errors can produce large errors when detections are mapped to pitch coordinates, especially at the far end where perspective foreshortening is strong.
    \item Lighting variation, camera shake, player occlusion, and inconsistent recording conditions reduce reliability in real grounds compared to controlled broadcast environments.
\end{enumerate}

Therefore, the central problem addressed by this project is whether a single-phone-camera workflow, supported by pitch calibration and lightweight computer-vision techniques, can produce a sufficiently consistent and understandable LBW review aid for ordinary cricket grounds.

\section{Objectives}

\subsection{General Objective}

To design and implement a low-cost cricket decision-support system that uses a single camera view to analyze a delivery, estimate the ball path in calibrated pitch coordinates, and assist LBW review.

\subsection{Specific Objectives}

\begin{enumerate}
    \item Study existing decision-review systems and relevant literature on monocular sports tracking.
    \item Develop a user-guided calibration process based on pitch corners and stump reference points.
    \item Detect and track the ball from delivery video using combined motion and colour cues.
    \item Map tracked pixel positions to pitch-plane coordinates using homography.
    \item Detect important events such as bounce and impact from the tracked trajectory using physics-based fitting.
    \item Estimate an LBW outcome using rule-based checks on pitching line, impact line, and projected stump contact.
    \item Present the result through an interactive 3D visualization that helps users interpret the decision.
\end{enumerate}

\section{Scope and Limitation}

\subsection{Scope}

The PocketDRS project encompasses the complete workflow from video capture to LBW decision:
\begin{itemize}
    \item Mobile application for video selection, trimming, and pitch calibration.
    \item Backend processing pipeline for ball detection, tracking, and 3D reconstruction.
    \item LBW decision engine implementing ICC Rule 36.
    \item Interactive 3D visualization of the trajectory.
    \item Local and cloud-based persistent storage of pitches and analyses.
\end{itemize}

\subsection{Limitation}

\begin{itemize}
    \item The system operates under umpire-POV camera placement; other angles may require retraining or recalibration.
    \item Accuracy degrades with heavy ball occlusion, poor lighting, or unstable camera mounting.
    \item The system is not ICC-certified and is designed for decision support rather than final authority.
    \item Monocular depth estimation remains fundamentally limited compared to multi-camera triangulation.
    \item Performance depends on proper pitch calibration; user error in marking reference points reduces decision reliability.
\end{itemize}

\chapter{Background Study and Literature Review}

\section{Background Study}

\textbf{Pinhole Camera Model and Intrinsic Calibration}

The pinhole camera model is the foundation for computer vision applications. It assumes light travels in straight lines through a small aperture and is captured on a planar image sensor. The intrinsic parameters (focal length and principal point) describe how a world point projects to image coordinates. Zhang's checkerboard calibration method~\cite{zhang2000} uses images of a known planar pattern to estimate intrinsics without requiring expensive calibration rigs. For PocketDRS, intrinsics are estimated from frame size and an assumed horizontal field of view (67 degrees), which is typical for smartphones.

\textbf{Extrinsic Calibration and PnP}

Extrinsic parameters (rotation and translation) describe the camera's pose in the world. The Perspective-n-Point (PnP) problem solves for this pose given known 3D world points and their 2D image projections. OpenCV's solvePnP~\cite{bradski2000} using ITERATIVE refinement computes the rotation vector and translation that minimize reprojection error. For cricket, pitch corners and stump positions are used as known landmarks.

\textbf{Homography and Pitch-Plane Mapping}

A homography is a $3 \times 3$ matrix that maps points between two coplanar surfaces~\cite{hartley2004}. For a flat pitch, the homography directly transforms pixel coordinates to pitch-plane coordinates in metres. It is computed from four point correspondences and can be refined using RANSAC to reject outliers.

\textbf{Projectile Motion and Trajectory Fitting}

A cricket ball in flight (ignoring air resistance, which is negligible over short distances) follows projectile motion: $X(t) = X_0 + V_x t$, $Y(t) = Y_0 + V_y t$, $Z(t) = Z_0 + V_z t - \frac{1}{2} g t^2$, where $g = 9.81$ m/s$^2$. This strong physical constraint allows recovery of smooth 3D trajectories from noisy detections. Bounce is modeled as a reflection of the vertical velocity component scaled by a coefficient of restitution (typically 0.55 for cricket).

\textbf{Kalman Filtering and State Estimation}

The Extended Kalman Filter (EKF) is used in monocular tracking to estimate ball state (position and velocity) over time. The process model applies projectile motion; the measurement model projects the 3D state to 2D image coordinates via the camera matrix. Although not used in the final implementation (RANSAC proved more robust for this application), EKF provides a principled framework for sensor fusion and uncertainty propagation.

\textbf{ICC Rule 36 and LBW Decision}

Law 36 (Leg Before Wicket) in cricket specifies when a batsman is out~\cite{icc}. Simplified, the ball must pitch in line with the wickets and then hit the batsman's leg in a line that would have hit the stumps. The LBW decision is umpire's call at the boundary; any outcome within approximately 25 mm of the edge is left to the umpire. PocketDRS implements these checks computationally.

\section{Literature Review}

Owens \textit{et al.}~\cite{owens2003} described the original Hawk-Eye ball-tracking system, which uses six to eight synchronized cameras and proprietary calibration to reconstruct ball trajectories to within a few millimetres. Their work is the inspiration and accuracy benchmark for PocketDRS, which trades multi-view precision for single-view affordability.

Zhang~\cite{zhang2000} introduced a flexible camera-calibration technique based on imaging a planar checkerboard, removing the need for expensive calibration rigs. This method underpins the intrinsic camera model assumed in this project.

Fischler and Bolles~\cite{fischler1981} proposed RANSAC, a robust estimator that fits a model to data containing outliers by repeatedly sampling minimal subsets and counting inliers. PocketDRS applies RANSAC both to homography estimation and to associating per-frame ball detections into a single trajectory.

Hartley and Zisserman~\cite{hartley2004} provide the standard reference on multiple-view geometry, homography, and projective reconstruction, which forms the theoretical basis for the single-view geometry used here.

Triggs \textit{et al.}~\cite{triggs2000} formalized bundle adjustment as the joint least-squares refinement of camera parameters and 3D structure; the same least-squares philosophy motivates the projectile-fit step in PocketDRS.

Ponglertnapakorn and Suwajanakorn~\cite{ponglertnapakorn2025} estimated 3D ball trajectories from a single monocular 2D track, showing that physical motion constraints make single-camera 3D ball reconstruction feasible. This is the closest prior work to PocketDRS and supports its core assumption.

Bochinski \textit{et al.}~\cite{bochinski2017} demonstrated high-quality multi-frame tracking by linking detections across frames using simple intersection-over-union association. PocketDRS follows the same detection-linking principle with a constant-acceleration motion model tailored to projectile flight.

\textbf{Synthesis}

The literature demonstrates that single-view sports analysis becomes feasible when the scene contains known geometry (the pitch) and the motion follows strong constraints (projectile motion). Homography-based mapping~\cite{hartley2004} and physics-based fitting are practical techniques for monocular reconstruction~\cite{ponglertnapakorn2025}. Prior work on monocular ball tracking shows that accuracy degrades gracefully when constraints are tight but remains useful for decision support. PocketDRS advances this area by demonstrating a complete end-to-end system combining mobile capture, cloud processing, and user-friendly visualization.

\subsection{Feasibility Analysis}

\textbf{Technical Feasibility}

The proposed system is technically feasible because it is built from mature and widely supported technologies. Flutter provides cross-platform mobile development with excellent performance. FastAPI provides lightweight and testable backend services with automatic OpenAPI documentation. OpenCV supports frame processing, tracking, and camera calibration. Firebase handles authentication and cloud persistence with real-time updates. The present implementation already demonstrates upload, calibration, server-side analysis, and result visualization on consumer devices. Although monocular video imposes accuracy limits, the required software stack is realistic for a student project and deployable on ordinary hardware.

\textbf{Operational Feasibility}

The operational workflow is simple enough for field use during testing and demonstrations. A user places a phone behind the bowler-end stumps, records a delivery, calibrates the pitch by tapping known points, trims the clip, and waits for the generated result. This procedure is repeatable, requires minimal training, and fits the needs of clubs, student projects, and coaching sessions.

\textbf{Economic Feasibility}

The project is economically feasible because it does not require specialized camera arrays, licensed broadcast systems, or custom hardware. It relies primarily on a smartphone, a tripod, and open-source libraries. Compared with professional DRS infrastructure (which costs hundreds of thousands of dollars), the expected deployment cost is extremely low, making the proposed system suitable for academic demonstration and low-budget cricket environments.

\section{System Design}

\subsection{Architectural Design}

PocketDRS follows a three-tier architecture:

\begin{itemize}
    \item \textbf{Presentation Tier (Mobile):} Flutter application with video capture, calibration UI, job submission, and result visualization via WebView (Three.js).
    \item \textbf{Application Tier (Backend):} FastAPI server with REST API endpoints for uploads, job status, and result retrieval, with asynchronous job processing via background tasks.
    \item \textbf{Data Tier (Cloud):} Firebase Authentication for user identity, Firestore for document storage, Google Cloud Storage for video files, and Google Cloud Run for scalable compute.
\end{itemize}

The backend processes videos asynchronously to avoid blocking HTTP responses. Progress is tracked in Firestore and polled by the client. The overall arrangement of the three tiers is shown in Figure~\ref{fig:architecture}.

\section{Algorithm Details}

The server-side analysis is made of five steps. Each is described below in plain terms: what it is given, what it produces, and how it works. The exact mathematical formulas are standard computer-vision techniques and are available in the references; the focus here is on \emph{what} each step does and \emph{why}.

\subsection{Step 1: Finding Where the Camera Was Standing}

\textbf{Given:} the four pitch corners the user tapped on the first frame, and the known real-world size of a cricket pitch.

\textbf{Produces:} the camera's position and the direction it was pointing.

Because a cricket pitch is always the same size (about 20~m long and 3~m wide), the system can compare the shape of the pitch as it appears in the image with its known real shape. From the difference between the two, it works out where the camera must have been standing and which way it was facing. This is a well-known technique in computer vision. One known problem is that the maths has two possible answers, one with the camera above the ground and one mirrored below it; the system simply rejects the below-ground answer, since a real camera is always above the pitch.

\subsection{Step 2: Finding the Ball in Each Frame}

\textbf{Given:} each video frame, and the user-selected ball colour.

\textbf{Produces:} a list of candidate ball positions in each frame, each with a confidence score.

The ball is found in two independent ways and the results are combined:

\begin{enumerate}
    \item \textbf{By motion.} The system compares each frame with a model of the still background and keeps only the parts that are moving. It then looks for ball-shaped moving objects. Importantly, it accepts the ball in two forms: as a small \emph{round blob} (a slow or distant ball), and as a short \emph{blurred streak} (a fast ball). A fast cricket ball filmed at 30 frames per second smears into a streak rather than a clean circle, so looking only for circles would miss it.
    \item \textbf{By colour.} The system also looks for regions matching the ball's colour (for example, the red of a cricket ball).
\end{enumerate}

When a motion detection and a colour detection agree on the same spot, the system is more confident that it has found the real ball.

\paragraph{Ignoring fixed clutter.}
Before the next step, the system removes any detection that keeps appearing in the \emph{same} place throughout the clip. A real ball only passes through any given spot once or twice; a red or round object that sits still the whole time (a logo, a helmet, a bag, a bat handle) is not the ball. Any candidate that appears in more than 30\% of the frames at the same location is treated as background and discarded. This is the single most important fix for real-world handheld videos: without it, the next step would happily draw a ``ball path'' through a cluster of fixed clutter.

\subsection{Step 3: Joining the Detections into One Ball Path}

\textbf{Given:} the list of candidate ball positions from Step 2.

\textbf{Produces:} a single smooth path through the frames, plus a count of how many detections fit it and how well.

The detections from Step 2 include the real ball plus some false positives. This step finds the one consistent path among them. It takes the early frames as a starting point, tries connecting pairs of detections, and for each guess checks how many of the later detections fall close to the path a real ball would follow (a ball moves in a smooth curve, slowing slightly and curving under gravity). The guess that explains the most detections wins.

Two design choices make this reliable for phone video shot from behind the stumps. First, the ball is moving almost straight \emph{towards} the camera, so it does not appear to move much across the screen; the test therefore looks at total movement over the whole clip rather than frame-to-frame speed. Second, if the best path found only covers a tiny part of the image, it is rejected as leftover clutter, and the system honestly reports ``no clear ball trajectory'' instead of inventing a decision.

\subsection{Step 4: Rebuilding the Ball Path in Real-World 3D}

\textbf{Given:} the ball path on the image (Step 3) and the camera position (Step 1).

\textbf{Produces:} the ball's path in real-world 3D coordinates (in metres).

A single camera cannot directly see depth, so the system estimates how far away the ball is in each frame from how big it looks: a cricket ball is always 0.036~m in radius, so a smaller ball in the image means a more distant ball. Combining that distance with the camera position gives a rough 3D point for each frame. These rough points are then fitted to the physics of a thrown object: a ball in flight follows a smooth curve shaped by gravity. Fitting this physical curve smooths out the measurement noise and gives the final 3D path. The path is also anchored using the point where the ball bounces on the ground, which is the most reliable single point to pin down.

\subsection{Step 5: Making the LBW Decision (ICC Rule 36)}

\textbf{Given:} the 3D bounce point, the 3D point where the ball hit the pad, and the predicted point where the ball would have reached the stumps.

\textbf{Produces:} a decision (OUT, NOT OUT, or UMPIRE'S CALL) with a plain-language reason and a confidence score.

The decision follows the three standard LBW checks from the Laws of Cricket. The key real-world measurements it uses are the width of the stumps, the radius of the ball (0.036~m), and a small ``umpire's call'' margin of 0.025~m around the edge of the stumps.

\begin{enumerate}
    \item \textbf{Did it pitch in line?} If the ball bounced outside the line of leg stump, the decision is NOT OUT (``pitched outside leg'').
    \item \textbf{Did it hit the pad in line?} If the ball struck the pad outside the line of the stumps, the decision is NOT OUT.
    \item \textbf{Would it have hit the stumps?} The predicted path is carried on to the stumps. If it would clearly miss, the decision is NOT OUT (``missing stumps'').
\end{enumerate}

If all three checks pass, the ball would have hit the stumps. If the predicted path passes within the small 0.025~m margin of the stump edge, the result is UMPIRE'S CALL (too close to be certain); otherwise it is OUT. The confidence score is higher when the 3D path was a tight, clean fit and lower when it was noisy.

\textbf{Backend Processing Modules}

\begin{itemize}
    \item \textbf{Video Decoder:} OpenCV-based frame extraction with support for multiple codecs and automatic rotation correction based on video metadata.
    \item \textbf{Ball Detector:} Combined motion (MOG2) and colour (HSV) detection with confidence weighting, plus ROI masking to reduce false positives from players and spectators.
    \item \textbf{Trajectory Finder:} RANSAC-based association linking per-frame detections into a single smooth ball trajectory with constant-acceleration fitting.
    \item \textbf{Calibration Module:} Computes homography from user-provided pitch corners using OpenCV's findHomography with RANSAC, and validates calibration quality via reprojection error.
    \item \textbf{Camera Pose Solver:} solvePnP with pitch corners and optional stump points to recover camera extrinsics (rotation and translation).
    \item \textbf{3D Reconstruction:} Back-projects image detections to 3D using depth-from-size, then fits projectile motion to produce a smooth world-coordinate trajectory.
    \item \textbf{LBW Decision Engine:} Rule-based checks for pitching line, impact line, and hitting stumps, producing OUT/NOT OUT/UMPIRES CALL with confidence.
    \item \textbf{Job Manager:} Asynchronous task queue managing video processing, with progress updates written to Firestore.
\end{itemize}

\subsection{Robustness and Edge-Case Hardening}
\label{sec:hardening}

Beyond the core algorithms, a dedicated review pass was carried out to find and fix the ways the system could crash or give a wrong answer on unusual input. Real videos and real users do not behave like clean test data, so this hardening work matters as much as the main pipeline. The fixes, in plain terms:

\begin{itemize}
    \item \textbf{Truncated video files.} Some video files claim to be longer than they really are (a wrong length field in the file header, common in trimmed or shared clips). The video reader now detects when it can no longer decode new frames and stops there, analysing the real frames it has instead of padding the analysis with repeated stale frames.
    \item \textbf{Bad pitch calibration.} If the four tapped pitch corners are all the same point, lie on a straight line, or do not form a proper four-sided shape, the calibration is now rejected immediately with a clear message, both in the app (before saving) and on the server, instead of failing confusingly much later.
    \item \textbf{Invalid numbers.} Negative pitch dimensions, an out-of-range camera field-of-view, and not-a-number values from the detector are now caught at the input boundary rather than silently flowing through the geometry and producing a meaningless result.
    \item \textbf{Network and session failures in the app.} If the job polling loop loses the connection it now stops with the real error after a few retries instead of polling silently for minutes; a genuine server-side job failure is reported immediately; an expired sign-in token produces a clean ``please sign in again'' message; and the video player resource is always released, even if loading fails midway.
    \item \textbf{3D viewer safety.} Non-finite numbers in a trajectory are coerced to safe values before being sent to the 3D view, and if the viewer's scripts never finish loading the user sees a clear error instead of a permanently blank screen.
\end{itemize}

Together these changes mean the system either produces a correct result or fails clearly and safely: it does not crash, and it does not invent a confident-but-wrong answer.

\subsection{Synthetic Validation}
\label{sec:synthval}

A controlled synthetic harness was built to validate the pipeline against delivery scenarios with known ground truth. OpenCV renders a textured pitch trapezoid, crease lines, and a red ball following a physically simulated projectile (gravity $9.81$ m/s$^2$, vertical restitution $0.55$). The synthetic camera is placed behind the striker at $(-3.5, 0, 1.8)$ m looking at $(10, 0, 0)$, with intrinsics $f_x = f_y = 900$, $c_x = 540$, $c_y = 960$ giving a $61.9^{\circ}$ horizontal field of view. Output is $1080 \times 1920$ portrait at 60 fps, matching a phone shot.

The harness sweeps three release speeds (medium-pace $18$, $24$, and $30$ m/s) across eleven release-line presets (clearly wide of off, off-stump line, middle stump, leg-stump line, outside leg, and going-down/wide deliveries) and two release angles, producing $66$ ground-truth-labelled scenarios.

\paragraph{Per-stage results.}
\begin{itemize}
    \item \textbf{Camera calibration.} PnP reprojection error $0.00$ px on every scenario; the recovered camera centre matches ground truth exactly to three decimal places. The fix that resolved this was disambiguating the twofold planar-PnP mirror by requiring the camera to be above the pitch.
    \item \textbf{Ball detection and trajectory association.} The combined MOG2 motion plus HSV-red colour detector finds $60$--$130$ candidates per scenario; the RANSAC association recovers $60$--$70$ inliers per delivery (typically $\sim 95\%$ of in-pitch detections), with image-space RMS in the $10$--$25$ px range.
    \item \textbf{3D reconstruction.} The bundle-adjusted projectile fit, augmented by an image-space bounce anchor that back-projects the bounce pixel onto the calibrated pitch plane, returns smooth parabolic trajectories that traverse the full pitch length. Physical bounds on the optimiser (release height $\in [0, 3.5]$~m, downward $v_z$, capped forward speed) eliminate the unphysical local minima that earlier bundle adjustment occasionally fell into.
    \item \textbf{LBW decisions.} Across the 66-scenario sweep the decision engine returns the correct verdict on $53/66$ deliveries ($80.3\%$ overall). Per class:
    \begin{itemize}
        \item OUT: $16/16$ ($100\%$).
        \item NOT OUT: $36/44$ ($81.8\%$).
        \item UMPIRE'S CALL: $1/6$ ($16.7\%$, see discussion below).
    \end{itemize}
\end{itemize}

\paragraph{Sample rendered decisions.}
Figure~\ref{fig:cases} (shown earlier, in the Interface Design section) gives real Hawk-Eye viewer renders of one scenario from each verdict class in this sweep: a leg-stump-line delivery (OUT), a delivery going down the leg side (NOT OUT), and a marginal leg-stump delivery (UMPIRE'S CALL). In each, the recovered blue trajectory starts at the bowler's release point, descends, bounces on the pitch, and the verdict follows from where the path meets the stump plane.

\paragraph{Where accuracy degrades.}
All $13$ failed verdicts are concentrated in two regimes that are genuinely difficult for monocular geometry:
\begin{enumerate}
    \item Fast deliveries ($30$ m/s) tracked over very few frames, where radius-from-size noise dominates depth resolution; and
    \item ``Leg marginal'' close calls where the recovered impact-Y differs from ground truth by $5$--$20$ cm and shifts the verdict from UMPIRE'S CALL to NOT OUT.
\end{enumerate}
Both reflect the inherent precision floor of single-camera depth-from-size on a ball travelling along the camera optical axis. Real Hawk-Eye uses six to eight synchronised cameras for exactly this reason; this project documents the monocular feasibility envelope rather than claiming broadcast-grade accuracy. Within the OUT class the system matches ground truth perfectly, which is the most consequential decision for an umpire to support.

\paragraph{What went wrong, in plain terms.}
Four separate problems were found, and only one of them was a software bug:

\begin{enumerate}
    \item \textbf{The video's length information was wrong.} The video file said it was 7 seconds long, but only 3.25 seconds of it could actually be read (98 frames). This is common with clips that have been trimmed or shared online; the editor does not always rewrite the file header. The video content itself was fine; only the header was wrong. This \emph{was} a real software gap: the pipeline trusted the header and tried to read frames that did not exist, padding the analysis with repeated stale frames. It was fixed (see Section~\ref{sec:hardening}).
    \item \textbf{The camera was too low.} The camera was behind the stumps (correct direction) but only at chest height. The bowler runs up and delivers right past the camera, so his body blocks 20--36\% of the view down the pitch, and it blocks it during exactly the moment the ball is in flight. Professional ball-tracking cameras are mounted high up, above head height, so they can see over the bowler. A low camera simply cannot see the ball.
    \item \textbf{The clip was filmed through netting.} Green practice-net mesh crosses the whole frame, including the path the ball travels.
    \item \textbf{The background was busy.} Swaying nets, trees, and two moving people (the bowler and the batter) create dozens of small moving objects every frame. The real ball is just one faint, low-contrast, motion-blurred object lost among them.
\end{enumerate}

\paragraph{Why this is a good result, not a failure.}
Because of the fixed-clutter removal and the path-length check added to the detector (Steps~2 and~3), the pipeline did the \emph{safe} thing: it reported ``no clear ball trajectory found'' instead of inventing a confident-but-wrong decision from background clutter. An earlier version of the system, before these checks, would have produced a fake verdict, the dangerous behaviour for any decision-support tool. A system that honestly says ``I cannot see the ball in this clip'' is trustworthy; one that guesses is not.

\paragraph{What a good test video looks like.}
The lesson from this clip is a simple checklist for recording a usable delivery: mount the phone \emph{high} (above head height) behind the bowler-end stumps so the bowler's body does not block the pitch; keep a clear line of sight with no netting between the camera and the pitch; record one continuous delivery with no edits or text overlays; and use the in-app tap calibration so the four pitch corners are marked precisely. With those conditions met (the same conditions the synthetic validation already simulates), the pipeline has everything it needs.

\paragraph{3D reconstruction and its honest limit.}
The recovered 2D track is then lifted into three dimensions. Because this is a single, uncalibrated handheld clip, the lift uses depth-from-apparent-size: the known physical radius of a cricket ball ($\approx 36$\,mm) against its measured pixel radius gives the ball's distance from the camera in each frame (estimated intrinsics $f_x = f_y \approx 816$\,px for the $67^{\circ}$ field of view). The right panel of Figure~\ref{fig:test3_path} shows the resulting 3D arc and Figure~\ref{fig:test3_pixel} the underlying pixel trajectory and the shrinking apparent-size profile that drives the depth estimate. The arc has the correct \emph{shape}: a flighted off-break that loops and descends onto the batter. Its absolute \emph{scale}, however, is monocular-ambiguous: a single camera on a short coaching net cannot pin down true distance, so the recovered depths ($1.4$--$2.4$\,m) and the implied average speed ($\approx 20$\,km/h) are order-of-magnitude estimates, not calibrated measurements. This is the same single-camera depth-from-size precision floor identified in the synthetic sweep (Section~\ref{sec:synthval}), and it is precisely why broadcast Hawk-Eye uses six to eight synchronised cameras.

\begin{figure}[H]
    \centering
    \includegraphics[width=0.95\textwidth]{figures/test3_pixel_track.png}
    \caption{Left: the ball's pixel trajectory on \texttt{test3.mp4}, coloured by time from release. Right: the ball's apparent radius shrinking through the flight, the depth cue used by the monocular 3D lift.}
    \label{fig:test3_pixel}
\end{figure}

\paragraph{Why this clip matters.}
Taken together, the two real clips bracket the system's real-world envelope. The first clip is the negative case: when the ball is hidden (low camera, netting, clutter) the pipeline refuses to invent a verdict. This clip is the positive case: when the ball stays in view, the same unchanged pipeline recovers a clean, smooth, full-flight 2D track and a plausibly-shaped 3D arc end-to-end on real handheld phone footage, while remaining honest about the metric precision a single phone camera can and cannot deliver. That combination, a usable track when the conditions are met and a safe refusal when they are not, is the behaviour a decision-support tool should have.

\chapter{Conclusion and Future Recommendations}

\section{Conclusion}

PocketDRS successfully demonstrates that a low-cost, single-phone-camera cricket decision-review system is technically feasible and operationally practical. The project delivers a complete end-to-end workflow from video capture to explainable LBW decision, integrating computer vision, 3D reconstruction, physics modeling, and domain-specific decision logic.

Key achievements include:
\begin{enumerate}
    \item A user-guided pitch calibration workflow that requires no specialized equipment or training.
    \item A combined motion and colour ball detector robust to real-world noise and false positives.
    \item RANSAC-based trajectory association that extracts a single ball path from cluttered per-frame detections.
    \item Physics-constrained monocular 3D reconstruction with $0.5$--$0.8$ m position accuracy on synthetic ground-truth trajectories.
    \item An ICC Rule 36 compliant LBW decision engine producing pitching, impact, and hitting-stump verdicts (out/not out/umpire's call) on both synthetic and real-world inputs.
    \item A mobile application and cloud backend demonstrating practical deployment with user-friendly visualization.
\end{enumerate}

The system is not ICC-certified and does not match the 3--5 mm accuracy of Hawk-Eye, but it is transparent, affordable, and suitable for decision support in low-resource cricket environments. The modular architecture enables future enhancements such as learned ball detection, multi-camera fusion, and on-device inference.

\section{Lesson Learnt/Outcome}

Several lessons emerged during development:

\begin{itemize}
    \item \textbf{Strong Constraints are Powerful:} Physics-based fitting (projectile motion) proved more effective than purely data-driven approaches in recovering clean 3D trajectories from noisy 2D detections. Domain knowledge is a feature, not a limitation, in research-scale systems.
    \item \textbf{User Calibration is Critical:} The quality of the system output is highly sensitive to calibration accuracy. A small error in marking pitch corners produces large errors at the far end. In future work, automatic landmark detection (stumps, crease lines) would reduce user burden and improve reliability.
    \item \textbf{Monocular Depth is Hard:} Depth-from-size (ball radius) is noisy when the ball is small in the image. Physics fitting mitigates this by enforcing smooth trajectories, but the problem remains fundamentally under-constrained. Multi-camera fusion or stereo would be a natural next step.
    \item \textbf{Agile Development Works:} Incremental prototyping with regular testing and feedback enabled early course correction. Building in phases (calibration $\rightarrow$ detection $\rightarrow$ tracking $\rightarrow$ reconstruction $\rightarrow$ LBW) allowed each component to be validated independently.
    \item \textbf{User Experience Matters:} Even a technically sound system fails if users cannot easily understand how to use it. Feedback loops, diagnostic warnings, and visual explanations are essential for adoption.
\end{itemize}

\section{Future Recommendations}

\begin{enumerate}
    \item \textbf{Multi-Camera Fusion:} Add support for stereo or multiple phone cameras to directly triangulate 3D positions, eliminating reliance on depth-from-size and physics fitting. This would improve accuracy and robustness significantly.
    \item \textbf{Learned Ball Detection:} Train a YOLO-based detector on cricket videos to replace hand-crafted motion and colour filters. A learned detector would handle diverse lighting, ball colours, and occlusions more robustly.
    \item \textbf{Automatic Landmark Detection:} Detect stumps, crease lines, and pitch edges automatically using keypoint detection rather than requiring users to mark them. This would reduce calibration burden and improve consistency.
    \item \textbf{Real-Time Streaming:} Support live streaming during a match with real-time decision feedback rather than post-clip analysis. This requires optimizing the pipeline for latency and deploying on edge devices.
    \item \textbf{On-Device Inference:} Implement ball tracking and simple decision logic on the mobile device for offline operation and reduced latency. Larger models (3D reconstruction, LBW logic) can remain on the server.
    \item \textbf{Improved Bounce Modeling:} Incorporate air resistance (drag), spin, and complex bounce physics (energy loss, deformation). The current model is highly simplified and could be refined based on cricket-specific research.
    \item \textbf{Statistical Confidence Intervals:} Propagate uncertainty through the pipeline (calibration error, detection noise, fitting residuals) to produce principled confidence intervals on predictions rather than single-point estimates.
    \item \textbf{User Studies:} Conduct human-factors studies with umpires and players to evaluate whether the system's explanations are intuitive and whether decisions are trusted, and integrate the feedback into UI/UX design.
\end{enumerate}

\section*{Appendix B: Calibration UI Workflow}

The calibration workflow in PocketDRS is designed for simplicity and robustness:

\begin{enumerate}
    \item \textbf{Frame Selection:} The user selects any frame from the video where the pitch is clearly visible (typically the beginning of the clip).
    \item \textbf{Corner Marking:} The user taps the four pitch corners in order: striker-end-left, striker-end-right, bowler-end-right, bowler-end-left (clockwise). Visual feedback (circles and lines) confirms each tap.
    \item \textbf{Quality Feedback:} The system computes a homography from the four taps and projects it back to image space. If reprojection error is high, a warning prompts the user to re-calibrate.
    \item \textbf{Stump Marking (Optional):} If visible, the user taps the stump bases and optionally the tops. These refine the camera pose and improve depth accuracy.
    \item \textbf{Save:} The calibration is saved locally and to Firestore so the user can reuse it for future deliveries from the same ground.
    \item \textbf{Reuse:} On future videos from the same pitch, the user selects a saved calibration rather than re-calibrating.
\end{enumerate}

This workflow balances accuracy (tight constraints from known world points) with usability (minimal user effort). Typical calibration time is 30--60 seconds on a smartphone.

\section*{Appendix D: Representative Source Code}

The LBW decision engine that implements the three ICC Rule 36 checks is reproduced below from \texttt{server/app/pipeline/process\_job.py}.

\begin{lstlisting}[language=Python]
def _decide_lbw(*, bounce, impact, pred_y_at_stumps, pred_z_at_stumps,
                stump_x_m, fit_rms_m) -> dict:
    """ICC-Rule-36-flavoured LBW decision from a 3D reconstruction.
    World Y is centred on the stump line; +-WICKET_HALF_WIDTH is the
    stump width and +BALL_RADIUS_M each side is the touching tolerance.
    """
    checks = {"pitching_in_line": True, "impact_in_line": True,
              "wickets_hitting": False}
    reason_parts = []

    # Check 1: pitched in line (allow one stump width outside leg).
    if bounce is not None:
        _, by, _ = bounce
        leg_line_limit = WICKET_HALF_WIDTH_M + 0.1143
        if abs(by) > leg_line_limit:
            checks["pitching_in_line"] = False
            reason_parts.append(f"Pitched outside leg ({by*100:+.1f}cm)")

    # Check 2: impact in line (within stump line + ball radius, off side).
    if impact is not None:
        _, iy, _ = impact
        if iy > WICKET_GUARD_M + 0.05:
            checks["impact_in_line"] = False
            reason_parts.append(f"Impact outside off ({iy*100:+.1f}cm)")

    # Monocular depth noise is larger in Z than Y, so widen the vertical
    # umpire's-call band to one ball-diameter and keep Y at one radius.
    MARGIN_Y_UMP = BALL_RADIUS_M           # 3.6 cm
    MARGIN_Z_UMP = 2.0 * BALL_RADIUS_M     # 7.2 cm

    # Check 3: would it hit the stumps? (per-axis umpire's-call bands)
    decision, margin_text = "not_out", ""
    if pred_y_at_stumps is not None and pred_z_at_stumps is not None:
        hits_h = abs(pred_y_at_stumps) <= WICKET_GUARD_M
        hits_v = 0.0 <= pred_z_at_stumps <= DEFAULT_STUMP_HEIGHT_M + BALL_RADIUS_M
        if hits_h and hits_v:
            checks["wickets_hitting"] = True
            margin_y = WICKET_GUARD_M - abs(pred_y_at_stumps)
            margin_z_top = (DEFAULT_STUMP_HEIGHT_M + BALL_RADIUS_M) - pred_z_at_stumps
            in_band = (margin_y <= MARGIN_Y_UMP or
                       margin_z_top <= MARGIN_Z_UMP or
                       pred_z_at_stumps <= MARGIN_Z_UMP)
            margin_text = " umpires_band" if in_band else " hitting"

    if all(checks.values()):
        decision = "umpires_call" if "umpires_band" in margin_text else "out"
    return {"decision": decision, "checks": checks,
            "reason": "; ".join(reason_parts) or "Hitting stumps"}
\end{lstlisting}
